# Pillar Definitions — Blueprint Creator
# Nine documents, their exact format, required sections, and completion criteria

---

## How to use this file

This file defines what each of the nine Blueprint Creator documents must contain, how it should be structured, and what "complete" means for each. Blueprint Creator reads this at session start. Every document produced must match its definition here.

---

## Document 1 — Blueprint

**Purpose:** Establish the vision. Everything downstream derives from this.
**Primary consumer:** Architect.
**Depth required:** Enough clarity that the Architect can validate vision, scope, constraints, and success metrics without first asking what the product is trying to be.

**Required sections:**
1. **What this is** — one paragraph describing what the product does in plain English. No technical terms.
2. **Who uses it** — the person or people this is built for. What they need. What problem it solves for them.
3. **Goals** — three to five specific outcomes that define success. Each goal must be testable: "users can do X" not "the app is good."
4. **Scope** — what is included in the first version. Explicit. Named.
5. **Out of scope** — what is NOT included. Equally explicit. At least three items.
6. **Constraints** — time, budget, team size, platform requirements, or any hard limits.
7. **Success definition** — one paragraph: how will you know this worked? What is the measurable outcome after launch?

**Completion criteria:** All seven sections present. Goals are testable. Scope and out-of-scope are explicit and non-overlapping.

**Failure patterns to avoid:**
- Goals that cannot be verified ("the app feels fast")
- Scope that includes everything ("all features")
- No out-of-scope section

## Document 2 — Tech Stack

**Purpose:** Establish the technology choices with rationale. The Architect validates and finalizes these.
**Primary consumer:** Architect.
**Depth required:** Enough specificity that the Architect can validate or revise stack choices without reopening first-principles discovery on platform, hosting, persistence, or major framework direction.

**Required sections:**
1. **Primary recommendation** — the specific technology choices for this project: language, framework, database, hosting. One recommendation per category.
2. **Why this stack** — plain English rationale for each choice. One to two sentences per item. No jargon without translation.
3. **Alternatives considered** — exactly two alternatives per major choice, with a plain English comparison. Why the primary was chosen over each.
4. **Versions** — specific versions for every major dependency. No "latest."
5. **What each piece does** — a one-sentence plain English description of every technology named. "PostgreSQL is a database — a structured way to store and retrieve your data."
6. **Open questions for Architect** — any technology decisions that require architectural judgment. Flagged explicitly.

**Completion criteria:** All six sections present. Every technology has a plain English description. Alternatives are named with comparison. Versions are specified.

**Failure patterns to avoid:**
- Technology choices without rationale
- Alternatives listed without comparison
- "Latest" as a version
- Technical terms without plain English translation

## Document 3 — File and Directory Structure

**Purpose:** Establish the project's physical layout. Every folder named and explained.
**Primary consumer:** Architect.
**Depth required:** Enough structure that the Architect can turn the layout into canonical `project.md` file structure without inventing major directories or ownership boundaries.

**Required sections:**
1. **Complete directory tree** — the full folder structure, using tree notation. Every folder and key file included.
2. **Folder descriptions** — a plain English description of each folder's purpose. One to two sentences. "What goes here and why."
3. **Key files** — the most important files named with a one-line description of their role.
4. **What is generated vs authored** — which files the developer writes versus which are generated by tools. Clear distinction.
5. **Configuration files** — all configuration files named and described.

**Completion criteria:** Tree is complete. Every folder has a plain English description. Generated vs authored distinction is explicit.

**Failure patterns to avoid:**
- Empty folders with no description
- Folder names without explanation
- No distinction between generated and authored files

## Document 4 — Domain Model

**Purpose:** Establish the core concepts the system works with. What things exist. How they relate.
**Primary consumer:** Architect.
**Depth required:** Enough clarity that the Architect can validate entities, relationships, and business rules and produce a canonical schema direction without guessing missing concepts.

**Required sections:**
1. **Entities** — every major concept the system stores or works with. Named. Described in plain English: "A User is a person who has an account. They have a name, email, and password."
2. **Relationships** — how entities connect. Plain English first: "A User can have many Orders. An Order belongs to exactly one User."
3. **Business rules** — the rules that govern the data. "A User cannot place an Order without a verified email." Each rule named and stated clearly.
4. **Data shapes** — what each entity stores. For each entity: its attributes, their types in plain English ("a list of text values"), and which are required.
5. **Diagram description** — a plain English description of a diagram that would show the entities and their relationships. "Picture a circle labeled User connected by a line to a circle labeled Order. The line says 'places many.'"

**Completion criteria:** All entities named. Relationships explicit and directional. Business rules stated as rules, not descriptions. Diagram described in plain English.

**Failure patterns to avoid:**
- Technical database terms without translation ("foreign key" without explanation)
- Relationships stated in only one direction
- Business rules implied but not stated

## Document 5 — Requirements

**Purpose:** Establish what the system must do. Each requirement stated as a testable condition.
**Primary consumer:** Architect, then Execution Spec Gate.
**Depth required:** Enough behavioral specificity that the Architect can harden each requirement into the five-element contract and Spec Gate can later translate the validated output into tasks without inventing user-facing behavior.

**Required sections:**
1. **Functional requirements** — what the system does. Each requirement formatted as:
   - **Name**: a short label (REQ-F-001)
   - **Description**: what the system does, in plain English
   - **Done when**: one sentence describing the verifiable condition that proves this is working
2. **Non-functional requirements** — how the system performs. Each formatted the same way (REQ-NF-001 etc.).
3. **Priority** — for each requirement: must have / should have / nice to have
4. **Open questions** — any requirement that needs more information before it can be finalized

**Completion criteria:** Every functional requirement has a Done when condition. Every non-functional requirement has a measurable standard. Priorities assigned.

**Failure patterns to avoid:**
- Requirements without Done when conditions
- "The system should be fast" without a specific threshold
- Requirements that describe implementation rather than behavior

## Document 6 — API and Interfaces

**Purpose:** Establish how the system's parts communicate. Every connection point defined.
**Primary consumer:** Architect, then backend-oriented auxiliary review when needed.
**Depth required:** Enough detail that the Architect can validate interfaces, auth, inputs, outputs, and error behavior without needing follow-up questions about basic communication paths.

**Required sections:**
1. **API overview** — plain English description of what the API does and who uses it
2. **Endpoints** — for each endpoint:
   - Name and path
   - What it does in plain English
   - What it receives (inputs)
   - What it returns (outputs)
   - Error states: what can go wrong and what the response is
3. **Authentication** — how the system knows who is making the request
4. **External integrations** — any third-party services the system connects to, what they provide, and what happens if they are unavailable

**Completion criteria:** All endpoints named with inputs, outputs, and error states. Authentication described. External dependencies flagged with failure behavior.

**Failure patterns to avoid:**
- Endpoints without error states
- Authentication not described
- External services without fallback behavior named

## Document 7 — UI and UX Specification

**Purpose:** Establish what the user sees and how they interact with it.
**Primary consumer:** Architect, then UI Specialist when needed.
**Depth required:** Enough detail that downstream design review can evaluate flows, screen states, accessibility, and interaction expectations without first reconstructing the user journey.

**Required sections:**
1. **Screen inventory** — every screen or page in the product, named and described in one sentence
2. **User flows** — the key paths a user takes through the product. Described in plain English: "User opens app → sees login → enters credentials → lands on dashboard."
3. **Screen specifications** — for each screen:
   - What the user sees
   - What they can do
   - What happens in each action
   - Empty state (what shows when there is no data)
   - Error state (what shows when something goes wrong)
4. **Design principles** — three to five principles that define the feel of this product. "Simple. Never more than one action per screen."
5. **Accessibility requirements** — any accessibility standards that apply

**Completion criteria:** All screens inventoried. Key flows documented. Each screen has empty and error states. Design principles stated.

**Failure patterns to avoid:**
- Screens without empty states
- Flows that don't show what happens when something fails
- No design principles

## Document 8 — Infrastructure and Environment

**Purpose:** Establish how the system runs. Local, staging, and production.
**Primary consumer:** Architect, then Operator.
**Depth required:** Enough operational detail that the Architect can validate environment expectations and the Operator can later understand setup, deployment, and failure inspection without missing the basic run model.

**Required sections:**
1. **Local development setup** — step-by-step instructions to get the project running on a new machine. Written as commands and plain English explanations.
2. **Environment variables** — every variable the system needs. Name, purpose, and whether it is required or optional. No actual values — placeholders only.
3. **Database configuration** — how the database is set up, migrated, and seeded.
4. **Deployment** — where the system runs in production and how it gets there.
5. **Monitoring and logging** — what gets logged and where. What an operator checks when something is wrong.

**Completion criteria:** Local setup is step-by-step. Every environment variable named and described. Deployment destination named. Logging described.

**Failure patterns to avoid:**
- Setup instructions that skip steps
- Environment variables without descriptions
- "Deploy to cloud" without specifying which cloud or how

## Document 9 — Session and Decision Log

**Purpose:** Record everything decided during the Blueprint Creator session. The Architect reads this to understand the reasoning behind the draft.
**Primary consumer:** Architect.
**Depth required:** Enough reasoning trace that the Architect can distinguish confirmed decisions, open questions, and assumptions without re-asking the user for the history of the Blueprint Creator session.

**Required sections:**
1. **Decisions made** — each significant decision from the session. What was chosen and why.
   Format: "Decided: [what]. Reason: [why]. Considered but rejected: [alternatives]."
2. **Open questions** — questions that came up but were not resolved. Each flagged for the Architect.
   Format: "Open: [question]. Needs: Architect judgment."
3. **Assumptions** — things assumed to be true that were not explicitly confirmed.
   Format: "Assumed: [what]. Basis: [why this seemed reasonable]."
4. **Session summary** — one paragraph summarizing what was covered and where the project stands at the end of this session.

**Completion criteria:** All decisions logged. All open questions captured. Assumptions explicit. Session summary present.

**Failure patterns to avoid:**
- Decisions without rationale
- Open questions not flagged for the Architect
- Assumptions buried inside other documents rather than surfaced here
